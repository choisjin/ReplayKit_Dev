ReplayKit User Guide Test Automation Platform — Recording, Playback, and Verification in One

System Overview

ReplayKit is a test automation platform that records user interactions on Android and IVI (In-Vehicle Infotainment) devices, automatically plays them back while verifying results through screenshot-based image comparison (SSIM), and generates Excel reports.

ComponentTechnologyRole
BackendPython (FastAPI)Device control, recording/playback engine, image comparison
FrontendReact + TypeScript + Ant DesignUser interface (web browser)
Device ConnectionADB / TCP Socket / SerialAndroid, HKMC IVI, serial devices, etc.

Core Workflow

① Connect Device → ② Record Actions → ③ Set Expected Images → ④ Play Scenario → ⑤ Review Results & Report

Installation

Method 1: Using the Installer (Recommended)

  1. Run ReplayKit_Setup_1.0.exe → Allow administrator privileges
  2. Confirm the installation path (default: C:\ReplayKit)
  3. Select components:
    • ReplayKit Core — Required (includes embedded Python)
    • DLT Viewer SDK — Check if you use the DLT log viewer GUI (included in Standard)
    • Vimba X SDK — Check if you use Vision Camera (included in Full)
  4. After file copy completes, the environment setup window (setup.bat) opens automatically
  5. Python is included as embedded, so no separate installation is needed
  6. If Git is not installed — The Git installer will launch automatically:
    • Click Next with default settings to install
    • After installation, press any key to continue
  7. Python packages install automatically → Git repository initialized → Done
  8. Launch ReplayKit after installation completes
💡 Embedded Python The distribution includes Python 3.10 Embedded, so you do not need to install Python on your system separately. Running setup.bat automatically installs pip and required packages into the embedded Python. For development, you can also use system Python 3.10 + venv.

Method 2: Manual Installation (Development Environment)

# Initial setup in the project folder (one-time only)
setup.bat

# If embedded Python is not found, it automatically creates a venv using system Python 3.10
# Installs venv, Python packages, and Node packages

Running

Distribution Launch

Double-click ReplayKit.bat to launch the server management GUI.

Development Environment: Server Management GUI

python server.py

A Tkinter-based GUI window opens to manage both backend and frontend simultaneously. When the server is ready, the browser opens automatically.

Development Environment: Manual Launch

# Terminal 1: Backend
venv\Scripts\activate
python -m uvicorn backend.app.main:app --reload --port 8000

# Terminal 2: Frontend
cd frontend
npm run dev

After launch, open http://localhost:5173 in your browser to access the ReplayKit interface.

💡 Tip The frontend automatically proxies API requests to the backend, so no separate CORS configuration is needed.

Updates

ReplayKit can be updated in two ways:

Method 1: Update from the Web (Recommended)

  1. Click the Update button in the left menu
  2. Click OK in the confirmation dialog
  3. A "Server updating..." waiting screen appears
  4. The server automatically shuts down → runs git pull → restarts
  5. The page refreshes automatically when the server is ready

Method 2: Re-run ReplayKit.bat

Close the program completely and re-run ReplayKit.bat. On startup, git pull runs automatically to update to the latest version.

💡 User Data Preserved During Updates User data such as scenarios, test results, device settings, and path configurations are not tracked by git, so they remain unchanged during updates.

Device Management

The Device tab is the first screen of ReplayKit, where you register and manage devices for testing. Devices are divided into Primary Devices and Auxiliary Devices, and devices of the same type are grouped together for display.

💡 Device ID Convention Each device is assigned a unique ID in the format ModelName_Number. Example: Connected_Wide_1, CAN_2. Drag-and-drop within a group to reorder; numbers are automatically reassigned.

Primary Device

The target device for screen interaction and screenshot capture.

TypeConnectionDescription
ADB USB or WiFi Android device. Connect via USB or WiFi using IP:port.
Example: 192.168.1.100:5555
HKMC TCP Socket HKMC 6th generation IVI system. Connect via IP and TCP port (default 5000).
Screen type selectable: Front Center, Rear Left, Rear Right, Cluster
Vision Camera GigE Vision (UDP) GigE Vision industrial camera. Automatically detected via GVCP protocol during scan, displaying MAC address, current IP, and subnet information. Additional features available with Vimba X SDK installed.

Connecting a Device

  1. Click Add Primary Device or Add Auxiliary Device button
  2. For primary devices, select Project and Equipment Model. The Device ID prefix is determined by the model (e.g., GVM → GVM_1)
  3. Select a device auto-detected by Scan, or enter the address manually in the Manual Connect tab
  4. For auxiliary devices, select a Module; the module name becomes the Device ID prefix (e.g., CAN → CAN_1)
⚠️ ADB Device Preparation Settings → Developer Options → USB Debugging must be enabled on the Android device. For WiFi connection, run adb tcpip 5555 on the device first.

Vision Camera Connection (GigE Vision)

GigE Vision cameras connected to the network are automatically detected during scanning. The scan results display the camera's current IP along with the PC network interface IP.

  1. When a camera is detected during Scan, the MAC, model, and current IP are shown
  2. If the camera IP is on a different subnet from the PC, click the Change IP button
  3. In the ForceIP dialog, an IP matching the PC subnet is automatically suggested
  4. Click OK to change the camera IP; the system rescans automatically
  5. Now click Connect to populate the manual connection tab and complete the connection
💡 What is ForceIP? ForceIP is a GigE Vision standard feature for forcing IP configuration. When the camera is on a different subnet from the PC and cannot communicate, it changes the camera's IP to match the PC's subnet. This setting may reset when the camera is rebooted.

  • Name — Change the device display name
  • Baudrate — Change the serial device communication speed
  • Module — Change the auxiliary device's connected module

Per-Device Buttons

Buttons appear on the right side of each device entry based on its status:

  • Connect — Shown for disconnected devices. Click to attempt connection
  • Disconnect — Shown for connected devices. Click to disconnect while keeping the registration
  • Edit — Modify the device's name, address, settings, etc.
  • Delete — Completely remove the device registration (disconnects and renumbers automatically)

Batch Action Buttons

Batch action buttons are available at the top of the device list:

  • Connect All — Attempt to connect all registered devices
  • Connect Selected (N) — Connect only devices selected via checkbox
  • Disconnect All — Disconnect all devices
  • Disconnect Selected (N) — Disconnect only selected devices
💡 Default State on Server Start When the server starts, all devices are in Disconnected state. You must manually click Connect All or individual Connect buttons on the Device tab. Disconnecting disables auto-reconnect. Click Connect again to re-enable auto-reconnect.

Recording

The Recording tab lets you watch the device's live screen and record user interactions to create test scenarios. Each interaction is saved as a Step.

Starting a Recording

Non-Recording State (Recording Control Card)

  1. In the recording control card, select an existing scenario from the Scenario dropdown or click Create New to create a new one
  2. When clicking Create New, enter the scenario name in the modal. If a duplicate name exists, you can choose to overwrite or rename
  3. Optionally enter a Description
  4. Click the Start Recording button

Scenario Management Buttons

Management buttons appear next to the scenario dropdown (when a scenario is selected):

  • Copy — Copy the current scenario with a new name
  • Rename — Rename the scenario
  • Delete — Delete the scenario (confirmation required)

Recording State

When recording starts, the layout changes:

  • Top bar: Scenario name (read-only) + description input + ● Recording tag and Stop Recording button on the right
  • Manual step add card: Row 1 has target device + step type, Row 2 has parameter input + delay. + Add Step button in the top-right
  • Recorded step list: Steps are added in real time as they are recorded

Click/drag on the device screen to automatically record steps. Even during recording, you can use all editing features including step reordering, Wait insertion, and expected image capture.

⚠️ Save Confirmation When navigating to another page or closing the browser after stopping recording, a save confirmation dialog appears. Unsaved changes will be lost if you navigate away without saving.
💡 Continue Recording Selecting an already saved scenario from the dropdown loads its steps. Use the Continue Recording button to append new steps after the existing ones.

Adding Steps Manually

During recording or edit mode, use the Add Step area at the bottom of the screen. Select the step type, enter the required values, and click the add button.

Expected Image

To verify that each step executed correctly, set an Expected Image. During playback, the actual screenshot is compared with the expected image to determine Pass/Fail.

Capturing an Expected Image

  1. Click the 📷 camera icon on the right side of the step list
  2. A comparison mode selection popup appears:
    • Full Screen — Capture the current screen immediately
    • Single Crop — Opens the crop editor modal
    • Exclude Region — Opens the exclusion region editor modal
    • Multi Crop — Opens the multi-crop editor modal
  3. After capture, a tag like 📷 FULL or 📷 CROP appears on the right side of the step's second row
  4. Click the tag to preview the expected image (with box overlays for exclude/multi-crop modes)
  5. Click the ✕ button next to the tag to clear the expected image
💡 Note When Changing Comparison Mode Selecting a new comparison mode via the camera icon automatically resets the previous mode's data (crop regions, exclusion regions, etc.). Example: Switching from CROP to FULL deletes the existing ROI information.

Image Comparison Modes

Four modes are available for comparing expected images with actual screenshots:

ModeIconDescriptionUse Case
Full Screen FULL Compare the entire screen using SSIM When the entire screen must match (default)
Single Crop SINGLE_CROP Crop a specific region and use template matching When checking only specific UI elements like status bar or notifications
Exclude Region FULL_EXCLUDE SSIM comparison excluding specified regions When ignoring changing areas like clock or battery
Multi Crop MULTI_CROP Compare multiple regions individually (all must pass) When verifying multiple UI elements simultaneously

Editing Comparison Regions

  • Exclude Region Editor (scissors icon) — Drag on the expected image to designate exclusion areas (red)
  • Crop Region Editor — Drag to designate comparison areas (green)
  • ROI Settings — Specify the comparison target area in full screen mode
  • Each region can be modified by clicking it in the list or deleted with the X button
💡 Similarity Threshold The default similarity threshold is 0.95 (95%). Scores at or above this value are Pass; below is Fail. In Multi Crop mode, individual thresholds can be set for each crop.

Batch Device Switch

Click the ⇄ Device Switch button at the top of the step card to display all Device IDs used in the scenario. Select the replacement device from the dropdown next to each ID and click Apply to batch-change device_id and screenshot_device_id for all steps.

💡 Usage Example When importing a scenario created on another PC, or when replacing a device, batch-change the Device ID to enable immediate playback.

Hard Key Long Press

When holding an HKMC device hard key button, a red gauge fills up. After holding for 3+ seconds, the entire button turns red and is recognized as a Long Press. Releasing before the gauge fills registers as a Short Press.

Automatic Screen Size Adjustment

Prevents the issue where the screen fills the entire card on portrait devices (smartphones, etc.), hiding the slider. When Screen Region Crop or Hard Key Panel is activated, the screen area automatically shrinks to provide control space at the bottom. Coordinate conversion is calculated based on screen ratio, so accuracy is not affected.

💡 Striped Background The step list alternates background colors on odd/even rows for easy visual distinction between steps.

Wait Step

A Wait step pauses execution for the specified duration before proceeding to the next step. Clicking the W button or Wait add button presents 3 modes to choose from:

ModeDescriptionParametersExample
Basic Fixed wait for the specified time (ms) Wait time (ms) Wait 1 second after screen transition: 1000ms
Cycle Wait time increases with each repeat cycle Start (ms) + Interval (ms) Start 3000ms, interval 3000ms, 3 repeats:
Cycle 1: 3000ms, Cycle 2: 6000ms, Cycle 3: 9000ms
Random Wait a random time within the specified range Min (ms) ~ Max (ms) Range 0~10000ms: Different wait time each execution
💡 Cycle Mode Usage Useful when you need to gradually increase wait times in repeated tests. Example: Increasing request intervals in server load testing.

Conditional Jump

You can specify the next step to execute based on the verification result (Pass/Fail) of a step. This allows branching test flows, skipping on failure, and other logic.

SettingDescription
Jump on PassJump to the specified step on verification success (default: next step)
Jump on FailJump to the specified step on verification failure, or end (END)
⚠️ Infinite Loop Warning When setting conditional jumps, be careful not to create circular references that jump back to earlier steps.

Playback (Run)

Run a scenario to automatically operate the device and verify results.

⚠️ Device Connection Required All devices used in the scenario must be in Connected state before playback. An error message is displayed and playback does not start if any device is not connected. Connect them on the Device tab first.

Preflight Check and Device Mapping

Before playback starts, a preflight check automatically verifies that devices used in the scenario match the currently connected devices. If Device IDs differ or a device is not connected, a Device Mapping Change dialog appears:

  • Shows a comparison between scenario-recorded devices and currently connected devices
  • Allows remapping mismatched devices to other currently connected devices
  • Mapping changes apply only to that playback and do not modify the original scenario
  1. Click the Play button in the scenario detail title
  2. Device connection status is automatically verified
  3. Set the Repeat Count if needed (default: 1)
  4. Results are displayed in real time as each step executes:
    • Pass — Similarity at or above threshold
    • Warning — Near the threshold boundary
    • Fail — Similarity below threshold
    • Error — Execution error occurred

Sleep Mode Check Before Playback

The PC's sleep mode settings are automatically checked when playback starts. If sleep mode is enabled, a warning dialog appears:

  • Block Sleep and Play — Temporarily blocks sleep mode during playback. Automatically restored after playback completes.
  • Cancel — Cancels playback. You can disable sleep mode manually in Windows Settings and try again.
⚠️ Caution for Long Playback Sessions If the PC enters sleep mode, network connections may drop, causing playback to terminate abnormally. Use the Block Sleep option for long-running tasks like repeated playback.

Playback Controls

ButtonFunction
PauseWait after the current step completes
ResumeContinue paused playback
StopImmediately stop playback and save results up to the current point

Intermediate Result Saving During Repeated Playback

When repeat count is set to 2 or more, results are automatically saved after each cycle completes. In case of abnormal termination (sleep, network disconnection, etc.), results from completed cycles are preserved and viewable on the test results page.

💡 Auto-Stop on WebSocket Disconnect If the WebSocket connection between the browser and server is lost, playback stops automatically. Results from completed cycles are saved.

Group Management

Multiple scenarios can be organized into groups for sequential execution. Use the Group Management button at the top to create and manage groups.

  1. Create Group — Enter a group name and create
  2. Add scenarios to the group (multiple allowed)
  3. Drag to reorder scenarios
  4. Optionally set conditional jumps between scenarios:
    • Scenario A Pass → Jump to Scenario C
    • Scenario A Fail → End (END)

Group Playback Result Storage Structure

During group playback, screenshots for each scenario are stored under results/{result_folder}/{index}_{scenario_name}/screenshots/. The index represents the execution order within the group (starting from 0).

Tips & Troubleshooting

FAQ

Q: Device does not appear in scan

Checklist:
  • Android: Check that USB debugging is enabled
  • Android: Reconnect the USB cable and approve the permission popup on the PC
  • HKMC: Verify they are on the same network and the firewall is not blocking
  • Serial: Verify the driver is installed and no other program is occupying the COM port
  • Try restarting the ADB server

Q: Similarity score is low during playback

  • If there are time-varying areas (clock, battery, etc.), use Exclude Region mode
  • To check only specific elements rather than the entire screen, use Single Crop or Multi Crop
  • You can adjust the similarity threshold (default 0.95)
  • Differences may occur if the device resolution differs from when the recording was made

Q: I want to run scenarios on a different PC

  • Export to package the scenario as a ZIP file
  • Import the ZIP file on the other PC
  • During playback, select the current PC's device in the Device Mapping step

Q: Server connection was lost

  • A "Connecting to server..." message appears at the top of the screen
  • Automatic reconnection is attempted; it recovers automatically when the server is back up
  • If unresolved, re-run server.py

Q: I want to adjust the wait time between steps

Change the delay_after_ms value in each step's Edit Command. The default is 1000ms (1 second); increase it if screen transitions are slow.

Q: "Server is already running" message appears

ReplayKit automatically detects duplicate instance execution. If an instance is already running, a dialog appears, allowing you to terminate the existing process and restart.

Automatic Server Management Features

  • Orphan result folder cleanup — On server startup, incomplete result folders without a result.json file are automatically deleted.
  • Date-based log storage — Server logs are managed as date-based files; logs older than 7 days are automatically deleted.

Module Guide

Modules are extension features callable from scenarios via module_command steps. They can be added as auxiliary devices on the Device page or inserted directly as steps during scenario editing.

CMD — Command Execution Module

A module that executes Windows system commands. Used for launching apps, process management, and system status checks during testing.

Connection Setup

No separate connection setup is required. Simply select CMD as the module to start using it.

Functions

FunctionDescriptionParametersReturn Value
Run Execute command and wait for completion (blocking) command — Command to execute
timeout — Max wait time in seconds (default 300) 		Execution output (stdout + stderr)
RunBackground Execute in background (non-blocking) command — Command to execute Process ID in PID:12345 format
Kill Terminate a background process pid — Process PID to terminate Termination result message
ListBackground List running background processes (none) List of alive PIDs (comma-separated)

Usage in Scenarios

Differences by step type:
  • cmd_send — Executes the command and ignores the result. For cases where only execution is needed, like launching apps or copying files
  • cmd_check — Executes the command and compares the output with an expected value. Set expected and match_type (contains / exact / regex)
  • module_command — Directly calls CMD module functions. For advanced features like RunBackground or Kill

Examples

PurposeStep TypeConfiguration
Launch app cmd_send command: start notepad.exe
Ping verification cmd_check command: ping -n 1 192.168.1.1
expected: TTL=, match_type: contains
Background execution module_command module: CMD, function: RunBackground

WaitLog vs StartMonitor:
  • WaitLog — Blocking: Pauses step execution until the keyword appears. Use for definitive verification.
  • StartMonitor — Non-blocking: Proceeds to the next step immediately while monitoring in the background. Check later with GetMonitorResult.
  • In the step add UI, selecting WaitLog and checking "Background monitoring" automatically switches to StartMonitor.
Typical DLT verification scenario flow:
  1. Add DLTViewer module as an auxiliary device (configure ECU IP)
  2. Log reception starts automatically on connection
  3. ClearLogs to clear the buffer
  4. Perform test actions (screen touch, app launch, etc.)
  5. WaitLog to verify the expected DLT log appears
