ReplayKit 사용자 가이드 테스트 자동화 플랫폼 — 녹화, 재생, 검증을 하나로

시스템 개요

ReplayKit은 Android 및 IVI(차량 인포테인먼트) 디바이스에서 사용자의 조작을 녹화하고, 이를 자동으로 재생하면서 스크린샷 기반 이미지 비교(SSIM)로 검증한 뒤 Excel 리포트를 생성하는 테스트 자동화 플랫폼입니다.

구성 요소기술역할
백엔드Python (FastAPI)디바이스 제어, 녹화/재생 엔진, 이미지 비교
프론트엔드React + TypeScript + Ant Design사용자 인터페이스 (웹 브라우저)
디바이스 연결ADB / TCP 소켓 / 시리얼Android, HKMC IVI, 시리얼 장치 등

핵심 워크플로우

① 디바이스 연결 → ② 조작 녹화 → ③ 기대 이미지 설정 → ④ 시나리오 재생 → ⑤ 결과 확인 & 리포트

설치 방법

방법 1: 설치 프로그램 사용 (권장)

  1. ReplayKit_Setup_1.0.exe 실행 → 관리자 권한 허용
  2. 설치 경로 확인 (기본: C:\ReplayKit)
  3. 구성 요소 선택:
    • ReplayKit Core — 기본 (필수, 내장 Python 포함)
    • DLT Viewer SDK — DLT 로그 뷰어 GUI를 사용하는 경우 체크 (Standard에 기본 포함)
    • Vimba X SDK — Vision Camera를 사용하는 경우 체크 (Full 선택 시 포함)
  4. 파일 복사가 완료되면 환경 설정 창(setup.bat)이 자동으로 열립니다
  5. Python은 내장(Embedded) 방식으로 포함되어 있어 별도 설치가 필요 없습니다
  6. Git이 없는 경우 — Git 설치 프로그램이 자동으로 실행됩니다:
    • 기본 설정 그대로 Next를 클릭하여 설치하세요
    • 설치 완료 후 아무 키를 눌러 진행합니다
  7. Python 패키지 자동 설치 → Git 저장소 초기화 → 완료
  8. 설치 완료 후 ReplayKit을 실행합니다
💡 내장 Python (Embedded Python) 배포판은 Python 3.10 Embedded를 포함하고 있어 시스템에 Python을 따로 설치할 필요가 없습니다. setup.bat 실행 시 자동으로 내장 Python에 pip 및 필요 패키지를 설치합니다. 개발 환경에서는 시스템 Python 3.10 + venv를 사용할 수도 있습니다.

방법 2: 수동 설치 (개발 환경)

# 프로젝트 폴더에서 초기 설정 (최초 1회)
setup.bat

# 내장 Python이 없으면 시스템 Python 3.10 + venv를 자동 생성합니다
# venv 생성, Python 패키지, Node 패키지를 모두 설치합니다

실행 방법

배포판 실행

ReplayKit.bat을 더블클릭하면 서버 관리 GUI가 실행됩니다.

개발 환경: 서버 관리 GUI

python server.py

Tkinter 기반 GUI 창이 열리며, 백엔드와 프론트엔드를 동시에 관리할 수 있습니다. 서버가 준비되면 브라우저가 자동으로 열립니다.

개발 환경: 수동 실행

# 터미널 1: 백엔드
venv\Scripts\activate
python -m uvicorn backend.app.main:app --reload --port 8000

# 터미널 2: 프론트엔드
cd frontend
npm run dev

실행 후 브라우저에서 http://localhost:5173에 접속하면 ReplayKit 화면이 표시됩니다.

💡 팁 프론트엔드가 백엔드의 API 요청을 자동으로 프록시하므로, 별도의 CORS 설정이 필요 없습니다.

업데이트

ReplayKit은 Git 기반 자동 업데이트를 지원합니다. 서버 관리 GUI의 동기화+시작 버튼을 클릭하면:

  1. 로컬 변경사항 자동 초기화 (빌드 파일 충돌 방지)
  2. Git으로 최신 코드를 자동 다운로드 (git pull)
  3. 변경된 Python 패키지 자동 설치
  4. 서버 자동 재시작 후 브라우저 오픈
💡 업데이트 사용자 데이터 보존 시나리오, 테스트 결과, 디바이스 설정, 경로 설정 등 사용자 데이터는 git 추적 대상이 아니므로 업데이트 시 변경되지 않습니다.

디바이스 관리

ReplayKit의 첫 화면인 디바이스 탭에서 테스트에 사용할 장치들을 등록하고 관리합니다. 디바이스는 주 디바이스와 보조 디바이스로 나뉩니다.

주 디바이스

화면 조작과 스크린샷 캡처의 대상이 되는 디바이스입니다.

유형연결 방식설명
ADB USB 또는 WiFi Android 디바이스. USB로 연결하거나 IP:포트로 WiFi 연결.
예: 192.168.1.100:5555
HKMC TCP 소켓 HKMC 6세대 IVI 시스템. IP와 TCP 포트(기본 5000)로 연결.
화면 유형 선택 가능: Front Center, Rear Left, Rear Right, Cluster
Vision Camera GigE Vision (UDP) GigE Vision 산업용 카메라. 스캔 시 GVCP 프로토콜로 자동 감지되며, MAC 주소, 현재 IP, 서브넷 정보가 표시됩니다. Vimba X SDK 설치 시 추가 기능 사용 가능.

디바이스 연결하기

  1. 스캔 버튼 클릭 — ADB, HKMC, Vision Camera가 동시에 자동 감지됩니다
  2. 감지된 디바이스 목록에서 원하는 기기를 선택하고 연결 클릭
  3. 감지되지 않는 경우 수동 연결 탭에서 주소를 직접 입력할 수 있습니다
⚠️ ADB 디바이스 준비 Android 기기의 설정 → 개발자 옵션 → USB 디버깅이 켜져 있어야 합니다. WiFi 연결 시에는 adb tcpip 5555 명령이 먼저 실행되어 있어야 합니다.

Vision Camera 연결 (GigE Vision)

스캔 시 네트워크에 연결된 GigE Vision 카메라가 자동으로 감지됩니다. 감지 결과에 카메라의 현재 IP와 PC 네트워크 인터페이스 IP가 함께 표시됩니다.

  1. 스캔에서 카메라가 감지되면 MAC, 모델, 현재 IP가 표시됩니다
  2. 카메라 IP가 PC와 다른 서브넷에 있다면 IP 변경 버튼을 클릭합니다
  3. ForceIP 대화상자에서 PC 서브넷에 맞는 IP가 자동으로 추천됩니다
  4. 확인을 누르면 카메라 IP가 변경되고, 자동으로 다시 스캔됩니다
  5. 이제 연결 버튼을 클릭하면 수동 연결 탭에 값이 채워지며, 연결 버튼으로 완료합니다
💡 ForceIP란? GigE Vision 표준의 IP 강제 설정 기능입니다. 카메라가 PC와 다른 서브넷에 있어 통신이 불가능할 때, 카메라의 IP를 PC와 같은 서브넷으로 변경하여 연결할 수 있게 합니다. 이 설정은 카메라 재부팅 시 초기화될 수 있습니다.

보조 디바이스

주 디바이스를 보조하는 외부 장치들입니다. 시리얼 통신, 모듈 제어 등에 사용됩니다.

유형연결 방식설명
시리얼 COM 포트 / USB 시리얼 통신 장치. 보율(Baudrate) 설정 가능 (기본 115200)
모듈 플러그인 확장 모듈 (CAN 통신, 전원 제어 등). 동적으로 로딩되는 플러그인
벤치 UDP 벤치 장비. IP:UDP 포트로 연결

디바이스 작업

  • 이름 변경 — 디바이스의 표시 이름을 편집
  • 정보 수정 — 주소, 보율, 모듈 할당 등 변경
  • 연결 해제 — 디바이스 등록 해제
  • ADB 서버 재시작 — 연결 문제 발생 시 ADB 서버를 재시작
  • 재연결 — 연결이 끊어진 디바이스에 다시 연결 시도

녹화

녹화 탭에서는 디바이스의 실시간 화면을 보면서 사용자 조작을 기록하여 테스트 시나리오를 만듭니다. 각 조작은 하나의 스텝(Step)으로 저장됩니다.

녹화 시작하기

  1. 시나리오 이름을 입력합니다 (예: "설정_WiFi_연결_테스트")
  2. 필요시 설명을 추가합니다
  3. 대상 디바이스를 선택합니다 (주 디바이스 목록에서)
  4. 녹화 시작 버튼을 클릭합니다
  5. 디바이스 화면이 실시간으로 스트리밍되며, 화면을 클릭/드래그하여 조작을 기록합니다
  6. 모든 조작이 끝나면 녹화 중지를 클릭하여 시나리오를 저장합니다
💡 이어서 녹화 이미 저장된 시나리오를 불러와서 뒤에 스텝을 추가할 수 있습니다. 저장된 시나리오 불러오기 → 시나리오 선택 → 이어서 녹화

수동 스텝 추가

녹화 중이거나 편집 모드에서 화면 하단의 스텝 추가 영역을 사용합니다. 스텝 유형을 선택하고 필요한 값을 입력한 뒤 추가 버튼을 클릭합니다.

기대 이미지

각 스텝이 올바르게 실행되었는지 확인하려면 기대 이미지를 설정합니다. 재생 시 실제 스크린샷과 기대 이미지를 비교하여 Pass/Fail을 판정합니다.

  1. 스텝 목록에서 기대 이미지를 설정할 스텝을 찾습니다
  2. 📷 버튼을 클릭하면 현재 디바이스 화면이 전체 화면 기대 이미지로 자동 캡처됩니다
  3. 캡처된 이미지가 스텝 옆에 썸네일로 표시됩니다
  4. 필요 시 이미지를 클릭하여 확대하거나, 비교 모드를 변경할 수 있습니다
✅ 자동 캡처 기능 기대 이미지가 없는 스텝을 실행하면, 실행 결과 스크린샷이 자동으로 기대 이미지로 저장됩니다. 처음 시나리오를 만들 때 편리하게 사용할 수 있습니다.

이미지 비교 모드

기대 이미지와 실제 스크린샷을 비교하는 4가지 모드를 제공합니다:

모드아이콘설명사용 상황
전체화면 FULL 화면 전체를 SSIM으로 비교 화면 전체가 동일해야 할 때 (기본값)
단일크롭 SINGLE_CROP 특정 영역만 잘라서 템플릿 매칭 상태바, 알림 등 특정 UI 요소만 확인할 때
영역제외 FULL_EXCLUDE 지정 영역을 제외하고 SSIM 비교 시계, 배터리 등 변하는 영역을 무시할 때
멀티크롭 MULTI_CROP 여러 영역을 각각 비교 (모두 통과해야 Pass) 여러 UI 요소를 동시에 확인할 때

비교 영역 편집

  • 영역제외 편집 (가위 아이콘) — 기대 이미지 위에서 마우스로 드래그하여 제외할 영역(빨간색)을 지정
  • 크롭 영역 편집 — 비교할 영역(초록색)을 드래그하여 지정
  • ROI 설정 — 전체화면 모드에서 비교 대상 영역만 지정
  • 각 영역은 목록에서 클릭하여 수정하거나 X 버튼으로 삭제할 수 있습니다
💡 유사도 임계값 기본 유사도 임계값은 0.95 (95%)입니다. 이 값 이상이면 Pass, 미만이면 Fail로 판정됩니다. 멀티크롭 모드에서는 각 크롭별로 개별 임계값을 설정할 수 있습니다.

스텝 편집

녹화된 스텝은 언제든 수정할 수 있습니다. 각 스텝의 메뉴(⋯)에서:

  • 커맨드 수정 — 좌표, 텍스트, 명령어 등 파라미터 변경
  • 기대이미지 재촬영 — 현재 디바이스 화면으로 기대 이미지 교체
  • 이 스텝 테스트 — 해당 스텝만 단독 실행 + 검증하여 결과 확인
  • Wait 삽입 — 이 스텝 뒤에 대기 시간 삽입
  • 삭제 — 스텝 제거

좌표 수정이 필요한 스텝(tap, long_press, swipe)은 편집 모드에서 디바이스 화면을 직접 클릭/드래그하면 새 좌표가 자동으로 적용됩니다.

조건부 이동

스텝의 검증 결과(Pass/Fail)에 따라 다음 실행할 스텝을 지정할 수 있습니다. 이를 통해 테스트 흐름을 분기하거나, 실패 시 건너뛰기 등의 로직을 구현합니다.

설정설명
Pass 시 이동검증 성공 시 지정한 스텝으로 점프 (기본: 다음 스텝)
Fail 시 이동검증 실패 시 지정한 스텝으로 점프 또는 종료(END)
⚠️ 무한 루프 주의 조건부 이동을 설정할 때, 이전 스텝으로 돌아가는 순환 참조가 발생하지 않도록 주의하세요.

시나리오 관리

시나리오 탭에서는 녹화된 시나리오를 조회, 실행, 관리합니다.

시나리오 목록

저장된 모든 시나리오가 목록으로 표시됩니다. 각 시나리오에서 가능한 작업:

  • 재생 — 시나리오를 디바이스에서 실행
  • 복사 — 다른 이름으로 복제
  • 이름 변경 — 시나리오 이름 수정
  • 병합 — 여러 시나리오를 하나로 합침
  • 삭제 — 시나리오 제거 (확인 필요)

재생 (실행)

시나리오를 실행하여 자동으로 디바이스를 조작하고 결과를 검증합니다.

  1. 시나리오 옆의 재생 버튼을 클릭합니다
  2. 디바이스 매핑 확인 — 시나리오가 녹화된 디바이스와 현재 연결된 디바이스를 매핑합니다. 디바이스가 다르면 여기서 변경할 수 있습니다
  3. 필요 시 반복 횟수를 설정합니다 (기본 1회)
  4. 실행을 클릭하면 재생이 시작됩니다
  5. 각 스텝이 실행될 때마다 결과가 실시간으로 표시됩니다:
    • Pass — 유사도가 임계값 이상
    • Warning — 경계값 수준
    • Fail — 유사도가 임계값 미만
    • Error — 실행 오류 발생

재생 중 제어

버튼기능

재개일시정지된 재생을 계속

그룹 관리

여러 시나리오를 그룹으로 묶어 순차적으로 실행할 수 있습니다. 그룹 단위로 실행하면 시나리오 간의 조건부 이동도 가능합니다.

  1. 그룹 만들기 — 그룹 이름을 입력하고 생성
  2. 그룹에 시나리오를 추가 (여러 개 가능)
  3. 드래그로 시나리오 순서 변경
  4. 필요 시 시나리오 간 조건부 이동 설정:
    • 시나리오 A의 Pass → 시나리오 C로 점프
    • 시나리오 A의 Fail → 종료(END)
  5. 그룹 재생을 클릭하면 포함된 시나리오가 순차적으로 실행됩니다

내보내기 / 가져오기

내보내기 (Export)

시나리오와 그룹을 ZIP 파일로 패키징하여 다른 PC에 배포할 수 있습니다.

  1. 내보낼 시나리오/그룹을 선택
  2. 내보내기 클릭 → ZIP 파일이 생성됩니다
  3. 설정에서 저장 경로를 지정했다면 해당 경로에, 아니면 브라우저 다운로드로 저장됩니다

가져오기 (Import)

  1. 가져오기 클릭 → ZIP 파일 선택
  2. 미리보기 화면에서 포함된 시나리오 목록과 충돌 여부를 확인합니다
  3. 충돌 시 처리 방식을 선택합니다:
    • 건너뛰기 — 기존 시나리오 유지
    • 덮어쓰기 — 가져오는 시나리오로 교체
    • 이름 변경 — 새 이름으로 추가
  4. 적용을 클릭하면 가져오기가 완료됩니다

테스트 결과

테스트 결과 탭에서는 시나리오 재생 결과를 확인하고 리포트를 생성합니다.

팁 & 문제 해결

자주 묻는 질문

Q: 디바이스가 스캔에 나타나지 않아요

확인 사항:
  • Android: USB 디버깅이 켜져 있는지 확인
  • Android: USB 케이블을 다시 연결하고 PC에서 허용 팝업이 뜨면 승인
  • HKMC: 동일 네트워크에 있는지, 방화벽이 차단하지 않는지 확인
  • 시리얼: 드라이버가 설치되어 있는지, 다른 프로그램이 COM 포트를 점유하고 있지 않은지 확인
  • ADB 서버 재시작을 시도해보세요

Q: 재생 시 유사도가 낮게 나와요

  • 시간에 따라 변하는 영역(시계, 배터리 등)이 있다면 영역제외 모드를 사용하세요
  • 화면 전체가 아닌 특정 요소만 확인하고 싶다면 단일크롭 또는 멀티크롭을 사용하세요
  • 유사도 임계값을 조정할 수 있습니다 (기본 0.95)
  • 디바이스 해상도가 녹화 시와 다르면 차이가 발생할 수 있습니다

Q: 다른 PC에서 시나리오를 실행하고 싶어요

  • 내보내기로 시나리오를 ZIP 파일로 패키징
  • 다른 PC에서 가져오기로 ZIP 파일을 불러옵니다
  • 재생 시 디바이스 매핑 단계에서 현재 PC의 디바이스를 선택하면 됩니다

Q: 서버 연결이 끊어졌어요

  • 화면 상단에 "서버 연결 중..." 메시지가 표시됩니다
  • 자동 재연결이 시도되며, 서버가 다시 실행되면 자동으로 복구됩니다
  • 해결되지 않으면 server.py를 다시 실행하세요

Q: 스텝 실행 중간에 대기 시간을 조절하고 싶어요

각 스텝의 커맨드 수정에서 delay_after_ms 값을 변경합니다. 기본값은 1000ms (1초)이며, 화면 전환이 느린 경우 늘리면 됩니다.

모듈은 module_command 스텝을 통해 시나리오에서 호출할 수 있는 확장 기능입니다. 디바이스 페이지에서 보조 디바이스로 추가하거나, 시나리오 편집 중 스텝으로 직접 삽입할 수 있습니다.

CMD — 명령어 실행 모듈

Windows 시스템 명령어를 실행하는 모듈입니다. 테스트 중 앱 실행, 프로세스 관리, 시스템 상태 확인 등에 활용합니다.

연결 설정

별도의 연결 설정이 필요 없습니다. 모듈 선택 시 CMD를 선택하면 바로 사용 가능합니다.

제공 함수

함수설명파라미터반환값
Run 명령어 실행 후 완료까지 대기 (blocking) command — 실행할 명령어
timeout — 최대 대기 시간, 초 (기본 300) 		실행 출력 결과 (stdout + stderr)
RunBackground 백그라운드로 실행 (non-blocking) command — 실행할 명령어 PID:12345 형태의 프로세스 ID
Kill 백그라운드 프로세스 종료 pid — 종료할 프로세스 PID 종료 결과 메시지
ListBackground 실행 중인 백그라운드 프로세스 목록 (없음) 살아있는 PID 목록 (쉼표 구분)

시나리오에서의 사용

스텝 유형별 차이:
  • cmd_send — 명령어를 실행하고 결과를 무시합니다. 앱 실행, 파일 복사 등 실행만 필요한 경우
  • cmd_check — 명령어를 실행하고 출력 결과를 기대값과 비교합니다. expected 값과 match_type (contains / exact / regex)을 설정
  • module_command — CMD 모듈의 함수를 직접 호출합니다. RunBackground나 Kill 등 고급 기능 사용 시

시나리오에서의 사용

모든 함수는 module_command 스텝으로 호출합니다. 가장 일반적인 활용은 DLT 로그 키워드 검증입니다:

사용 예시

목적함수설정
로그 키워드 대기 (블로킹) WaitLog keyword: BOOT_COMPLETE, timeout: 60
→ 60초 내 해당 키워드 출현 시 PASS, 스텝 진행 멈춤
AND 조건 검증 WaitLog keyword: APP_START navigation
→ 두 단어 모두 포함된 줄 검색
백그라운드 모니터링 StartMonitor keyword: ERROR, timeout: 120
→ 다음 스텝 즉시 진행, 별도 스레드에서 120초간 감시
모니터링 결과 확인 GetMonitorResult monitor_id: mon_1
→ PASS/FAIL/RUNNING 반환
로그 초기화 ClearLogs → 이전 로그를 비우고 새로운 검증 시작점 설정
GUI로 로그 시각화 LaunchViewer project_file: CCIC_DLT_VIEWER_PRJ.dlp
WaitLog vs StartMonitor:
  • WaitLog — 블로킹: 키워드가 나올 때까지 스텝 진행을 멈춥니다. 확실한 검증에 사용.
  • StartMonitor — 비블로킹: 다음 스텝으로 즉시 진행하면서 백그라운드에서 감시합니다. 나중에 GetMonitorResult로 확인.
  • 스텝 추가 UI에서 WaitLog 선택 후 "백그라운드 모니터링" 체크하면 자동으로 StartMonitor로 전환됩니다.
전형적인 DLT 검증 시나리오 흐름:
  1. DLTViewer 모듈을 보조 디바이스로 추가 (ECU IP 설정)
  2. 연결 시 자동으로 로그 수신 시작
  3. ClearLogs로 버퍼 초기화
  4. 테스트 동작 수행 (화면 터치, 앱 실행 등)
  5. WaitLog로 기대하는 DLT 로그가 나타나는지 검증

이미지 비교 모드 총정리

모드비교 방식판정 기준
FULL 전체 화면 SSIM (Structural Similarity) 전체 유사도 ≥ 임계값 → Pass
SINGLE_CROP 크롭 이미지로 실제 화면에서 위치 검색 (템플릿 매칭) → 해당 영역 SSIM 매칭 유사도 ≥ 임계값 → Pass
FULL_EXCLUDE 전체 화면 SSIM에서 지정 영역 제외 제외 후 나머지 영역 유사도 ≥ 임계값 → Pass
MULTI_CROP 복수 크롭 각각 독립 비교 모든 크롭의 유사도가 각각 임계값 이상 → Pass

결과 상태 코드

상태의미조건
Pass 검증 성공 유사도가 Pass 임계값 이상
Warning 경고 (주의 필요) 유사도가 Warning 임계값과 Pass 임계값 사이
Fail 검증 실패 유사도가 Warning 임계값 미만
Error 실행 오류 디바이스 연결 실패, 스크린샷 캡처 실패 등
